Delivery Harness Framework / DHF 当前技能路由

每个 DHF 阶段由哪个技能驱动。

MyCodexEnv 使用 project-lifecycle-harness 作为 Delivery Harness Framework (DHF) 的入口。DHF 先恢复运行时事实，探测 Codex 环境，识别当前生命周期阶段，再把任务路由到仓库专用 harness、gstack 专家技能、本地开发与验证技能，或确定性的运行时辅助脚本。辅助脚本提供机器可检查的执行层；技能负责认知编排。

路由规则

1. 运行时事实优先于记忆 先运行恢复和环境探测。聊天历史与 subconscious 输出只是线索，不是唯一事实来源。
2. 规划前必须通过需求门禁 如果范围或验收标准不稳定，先创建或验证需求产物，再进入架构规划。
3. 没有验证器，不允许并行写入 任何 worker 团队都必须通过 harness_agent_team.py validate，证明写入集合不重叠。
4. 完成声明必须有新鲜证据 声明完成前，记录 command、exit_code、key_output 和 timestamp。
5. 交接是运行时事件 在有意义的工作切片结束时，使用 harness_checkpoint.py append 记录命令、退出码、关键输出、时间戳和下一个安全任务。
6. 仓库不能强行定义全局沙箱真相 环境探测可以报告可观察到的沙箱和审批设置，但不能假装知道隐藏的 Desktop 线程状态。
